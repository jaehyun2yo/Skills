# Project Constitution Template — 프로젝트 원칙

## 문서 구조

```markdown
# [프로젝트명] — 프로젝트 원칙 (Project Constitution)

> 버전: 1.0 | 작성일: YYYY-MM-DD | 기반 문서: PRD v1.0

이 문서는 프로젝트 전반에 걸쳐 적용되는 불변 규칙과 제약 조건을 정의한다.
모든 설계, 구현, 리뷰는 이 문서의 원칙을 준수해야 한다.

---

- **예시**:
  - 준수: [좋은 예]
  - 위반: [나쁜 예]

예시 원칙들:

### CONST-XXX: AI 코드 생성 규칙
- [예: AI가 생성한 코드도 동일한 코딩 표준 적용]
- [예: AI 생성 코드는 반드시 리뷰 후 커밋]
- [예: 이 CONSTITUTION 파일을 AI 컨텍스트에 항상 포함]

### CONST-XXX: AI 구현 제약조건 (CONSTRAINTS 블록)
AI가 코드를 생성할 때 반드시 준수해야 하는 명시적 제약 상자.
모호한 가이드 대신 구체적 경계를 설정하여 AI의 "자신 있게 틀린 코드" 생성을 방지한다.

```
CONSTRAINTS:
- 새로운 public API 표면: [허용 범위 명시]
- 하위 호환성: [대상 버전 명시]
- 입력 검증 경계: [어디서 검증하는지 명시] — 중복 검증 생략
- 에러 전략: [구체적 예외 타입]으로 fail fast — silent swallowing 금지
- 언어 규칙: "should"/"may" 금지 → "must"/"must not"만 사용
- 조건문: 양쪽 분기를 항상 명시적으로 지정
```

### CONST-XXX: 안티패턴 DO NOT 목록
프로젝트에서 금지하는 구체적 패턴 목록. AI가 흔히 생성하는 문제 패턴을 선제적으로 차단한다.

```
DO NOT:
- 범용 Exception/Error catch 금지 — 구체적 예외 타입 사용
- 매직 넘버/문자열 하드코딩 금지 — 명명된 상수 사용
- console.log/print 디버깅 코드 커밋 금지
- any 타입 사용 금지 (TypeScript)
- 순환 의존성 금지
- 하드코딩된 시크릿/비밀번호 금지
- 빈 catch 블록 금지
- God Object 패턴 금지 (단일 파일 300줄 초과 시 분리)

## 작성 가이드

- **원칙은 구체적으로**: "좋은 코드를 작성한다" ✗ → "단일 함수는 50줄 이하, 하나의 책임만 가진다" ✓
- **근거를 붙인다**: 모든 원칙에 "왜"를 설명. 근거 없는 규칙은 지켜지지 않는다.
- **위반 예시 포함**: 준수/위반 예시가 있어야 모호함이 줄어든다.
- **프로젝트에 맞게 조절**: 위의 모든 섹션이 필수는 아니다. 1인 사이드 프로젝트에 코드 리뷰 정책은 불필요. 프로젝트 규모와 팀 구성에 맞게 선별.
- **변경에 보수적**: 이 문서는 자주 바뀌면 안 된다. "불변 규칙"이므로 신중하게 작성하고, 변경 시 영향 범위를 반드시 검토.
