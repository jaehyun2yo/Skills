# Coding Best Practices Reference

프로젝트 원칙(Constitution) 작성 시 참조하는 코딩 베스트 프랙티스 가이드.
프로젝트 성격에 맞게 선별하여 적용한다 — 모든 항목이 모든 프로젝트에 필수는 아니다.

---

## 1. 네이밍 컨벤션

### 원칙
- 서술적이고 검색 가능한 이름 사용
- 매직 넘버는 명명된 상수로 대체
- 접두사나 타입 정보를 이름에 포함하지 않음 (헝가리안 표기법 금지)
- 축약어보다 명확한 전체 단어 사용

### 언어별 기본 규칙

| 대상 | JavaScript/TS | Python | Go | Rust |
|------|-------------|--------|-----|------|
| 변수 | camelCase | snake_case | camelCase | snake_case |
| 함수 | camelCase | snake_case | PascalCase(공개)/camelCase(비공개) | snake_case |
| 클래스/타입 | PascalCase | PascalCase | PascalCase | PascalCase |
| 상수 | SCREAMING_SNAKE | SCREAMING_SNAKE | PascalCase | SCREAMING_SNAKE |
| 파일 | kebab-case | snake_case | snake_case | snake_case |
| 환경변수 | SCREAMING_SNAKE | SCREAMING_SNAKE | SCREAMING_SNAKE | SCREAMING_SNAKE |

## 2. 함수 설계

- **단일 책임**: 한 함수는 한 가지 일만 수행
- **크기 제한**: 50줄 이하 권장, 초과 시 분리 검토
- **최소 파라미터**: 3개 이하 권장, 초과 시 객체/구조체로 묶기
- **부작용 금지**: 입력을 받아 출력을 반환, 외부 상태 변경 최소화
- **플래그 파라미터 금지**: `doSomething(true)` 대신 별도 함수로 분리

## 3. 설계 원칙 (KISS / DRY / YAGNI / SOLID)

| 원칙 | 정의 | 적용 기준 |
|------|------|----------|
| **KISS** | 가능한 한 단순하게 | 3줄 유사 코드가 조기 추상화보다 낫다 |
| **DRY** | 모든 로직은 코드베이스 내 단일 표현 | 3번 이상 반복 시 추상화 검토 |
| **YAGNI** | 필요하지 않은 것은 만들지 않음 | 가상의 미래 요구사항을 위한 설계 금지 |
| **SRP** | 클래스/모듈은 변경 이유가 하나만 존재 | 파일이 300줄 이상이면 분리 검토 |
| **OCP** | 확장에 열려있고 수정에 닫혀있음 | 인터페이스/추상 클래스 활용 |
| **LSP** | 하위 타입은 상위 타입을 대체 가능 | 상속보다 합성 선호 |
| **ISP** | 클라이언트별 세분화된 인터페이스 | 거대 인터페이스 분리 |
| **DIP** | 구체가 아닌 추상에 의존 | 의존성 주입 패턴 |

## 4. 에러 처리

- **Fail Fast**: 잘못된 입력은 최대한 빨리 거부
- **구체적 예외**: 범용 `Exception` 대신 도메인별 예외 타입 사용
- **Silent Swallowing 금지**: 빈 catch 블록 금지, 최소한 로깅 필수
- **시스템 경계에서만 검증**: 내부 함수 간 중복 검증 불필요
- **복구 가능/불가능 구분**: 복구 가능한 에러는 처리, 불가능하면 즉시 전파

## 5. 보안 (OWASP Top 10 기반)

| 위협 | 방어 |
|------|------|
| **인젝션** (SQL, Command, XSS) | 파라미터 바인딩, 입력 이스케이프, CSP 헤더 |
| **인증 실패** | 검증된 라이브러리 사용, 비밀번호 해싱 (bcrypt/argon2) |
| **민감 데이터 노출** | TLS 강제, PII 마스킹, 로그에 시크릿 금지 |
| **접근 제어 실패** | 기본 거부(deny-by-default), RBAC/ABAC |
| **보안 설정 오류** | 기본 비밀번호 변경, 불필요한 기능 비활성화 |
| **취약 컴포넌트** | 의존성 자동 스캔 (npm audit, pip-audit) |
| **SSRF** | URL 화이트리스트, 내부 네트워크 접근 차단 |

**시크릿 관리 규칙**:
- 하드코딩 금지 → 환경 변수 또는 시크릿 매니저 사용
- `.env` 파일은 `.gitignore`에 반드시 포함
- AI 생성 코드에서 하드코딩된 시크릿 여부 반드시 검토

## 6. 성능

**정량적 기준 설정** (모호한 표현 금지):
- Bad: "빠르게 응답해야 한다"
- Good: "API 응답 시간 p95 < 200ms, p99 < 500ms"

**주요 지표**:
| 지표 | 측정 대상 | 예시 기준 |
|------|----------|----------|
| 응답 시간 | API latency p95/p99 | < 200ms / < 500ms |
| 처리량 | 동시 요청 | > 1000 req/s |
| 번들 크기 | 초기 로드 JS/CSS | < 200KB gzipped |
| 메모리 | 프로세스 RSS | < 512MB |
| DB 쿼리 | 단일 요청 당 쿼리 수 | < 10 queries |

**확장성**:
- 수평 확장 가능한 설계 (stateless 우선)
- 알려진 한계점을 명시 (예: "단일 DB 인스턴스로 10만 DAU까지")

## 7. 테스트

### 테스트 피라미드

| 유형 | 범위 | 속도 | 비율 |
|------|------|------|------|
| 단위 | 함수/클래스 단위 | 빠름 | 70% |
| 통합 | 모듈 간 연동 | 보통 | 20% |
| E2E | 전체 사용자 시나리오 | 느림 | 10% |

### 테스트 원칙
- 테스트당 하나의 동작만 검증
- 다른 테스트와 독립적, 순서 무관
- 프로덕션 코드와 동일한 코딩 표준 적용
- 엣지 케이스 반드시 포함: null, 빈 값, 경계값, 대용량

### 커버리지 기준
- 핵심 비즈니스 로직: 80% 이상
- 유틸리티/헬퍼: 90% 이상
- UI 컴포넌트: 주요 인터랙션 커버

## 8. AI 구현을 위한 스펙 작성 규칙

### 정확한 언어 사용
- "should" / "may" 금지 → "must" / "must not" 사용
- 양쪽 분기 명시:
  - Bad: "x > 0이면 A를, 아니면 B를 하라"
  - Good: "x > 0이면 A를 수행한다. x <= 0이면 B를 수행한다."

### 제약조건 블록 (CONSTRAINTS)
모든 스펙 문서에 포함:
```
CONSTRAINTS:
- 새로운 public API 표면은 [X]로 제한
- [version]과 하위 호환 유지
- 입력은 [boundary]에서 사전 검증됨 — 중복 검증 생략
- [SpecificException]으로 fail fast — silent swallowing 금지
```

### 안티패턴 DO NOT 목록
각 스펙에 "하지 말 것" 섹션을 포함:
```
DO NOT:
- 범용 Exception catch 금지
- 매직 넘버 하드코딩 금지
- console.log 디버깅 코드 커밋 금지
- any 타입 사용 금지 (TypeScript)
- 순환 의존성 금지
```

## 9. 문서화

- **자기 설명적 코드 우선**: 주석보다 코드 자체로 설명
- **주석은 "왜"만 작성**: "무엇"은 코드가 설명, "왜"만 주석
- **주석 처리된 코드 제거**: 버전 관리가 이력을 관리함
- **공개 API에 docstring/JSDoc 필수**
- **README 필수 항목**: 설치, 실행, 환경 변수, 기여 가이드

## 10. 피해야 할 코드 냄새

| 냄새 | 증상 | 해결 |
|------|------|------|
| 강직성 | 작은 변경이 연쇄 수정 촉발 | 인터페이스로 분리 |
| 취약성 | 변경이 무관한 영역을 깨뜨림 | 의존성 방향 정리 |
| 비이동성 | 코드 재사용 불가 | 모듈 경계 재설계 |
| 불필요한 복잡성 | 과도한 추상화 | YAGNI 적용 |
| 불필요한 반복 | 동일 로직 복사 | DRY 적용 |
| 불투명성 | 이해하기 어려운 코드 | 네이밍 개선, 함수 분리 |
| God Object | 하나의 클래스가 모든 것을 담당 | SRP 적용 |
| Feature Envy | 다른 클래스의 데이터를 과도하게 사용 | 메서드를 데이터 소유 클래스로 이동 |
