## 1. 테스트 전략 개요

### 1.1 테스트 목표
이 문서가 달성하려는 검증 목표. SRS의 모든 요구사항과 기능명세서의 모든 동작이 검증 가능한 테스트 케이스에 매핑되는 것.

### 1.2 테스트 범위
| 포함 | 제외 |
|------|------|
| [예: 핵심 비즈니스 로직] | [예: 서드파티 라이브러리 내부] |
| [예: API 엔드포인트] | [예: 외부 서비스 내부 동작] |
| [예: 사용자 인증 플로우] | [예: 인프라/네트워크 레벨] |

### 1.3 테스트 유형 및 도구

| 유형 | 목적 | 도구 | 대상 |
|------|------|------|------|
| 단위 테스트 | 개별 함수/모듈 | [예: Vitest] | 비즈니스 로직, 유틸리티 |
| 통합 테스트 | 컴포넌트 간 연동 | [예: Vitest + Supertest] | API 라우트, DB 연동 |
| E2E 테스트 | 사용자 시나리오 | [예: Playwright] | 핵심 사용자 플로우 |
| 성능 테스트 | 부하/응답 시간 | [예: k6] | API 엔드포인트 |
| 보안 테스트 | 취약점 검증 | [예: OWASP ZAP] | 인증, 입력 검증 |

### 1.4 테스트 환경
| 환경 | 용도 | DB | 외부 서비스 |
|------|------|-----|-----------|
| Local | 개발 중 단위/통합 | SQLite / In-memory | Mock |
| CI | PR 검증 | [예: PostgreSQL (Docker)] | Mock / Sandbox |
| Staging | E2E / 성능 | 프로덕션 복제본 | Sandbox |

**입력 데이터**:

**BDD 형식** (선택 — Gherkin):
```gherkin

```

## 7. CI/CD 통합

### 7.1 자동화 대상
| 유형 | 실행 시점 | 실패 시 동작 |
|------|----------|-------------|
| 단위 테스트 | 모든 커밋 | PR 머지 차단 |
| 통합 테스트 | PR 생성 시 | PR 머지 차단 |

| 성능 테스트 | 릴리스 전 | 경고 (수동 판단) |

### 7.2 테스트 실행 명령
```bash

npm run test:unit

# 통합 테스트
npm run test:integration

# E2E 테스트
npm run test:e2e

# 전체
npm run test
```

## 작성 가이드

- **모든 REQ에 TC가 있어야 한다**: 테스트 커버리지 매트릭스에서 빈칸이 있으면 안 된다. 커버리지 매트릭스가 이 문서의 핵심.
- **실패 케이스가 더 중요**: 성공 케이스 1개당 실패/엣지 케이스 2-3개가 적절. 버그의 대부분은 예외 상황에서 발생.
- **BDD는 선택**: Gherkin 형식은 비개발자와 소통할 때 유용하지만, 팀 구성에 따라 생략 가능.
- **자동화 가능성**: 각 테스트 케이스가 자동화 가능한지 표시. 수동 테스트가 필요한 항목은 그 이유를 명시.
- **성능/보안은 별도 섹션**: 기능 테스트와 섞지 않고 별도 관리. 각각 전문 도구가 다르기 때문.
