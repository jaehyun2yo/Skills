# E2E Testing Pipeline Upgrade Design

## Summary

Upgrade the existing declarative TDD rules ("write tests first") into a **procedural E2E testing pipeline** that actually writes E2E tests, runs them, and enforces fix loops — reducing manual user QA.

### Scope
- **project-setup**: E2E infrastructure scaffolding (SKILL.md, checklist.md, setup-templates.md)
- **team-setup**: Declarative rules → 6-phase procedural pipeline (SKILL.md — team-rules.md template content)
- **TaskCompleted hook**: 5-step E2E verification gate + auto fix loop (setup-templates.md Section B hook template)

### Key Decisions
| Decision | Choice | Rationale |
|----------|--------|-----------|
| Approach | Upgrade 2 existing skills + 1 hook template (no new skills) | Minimal change, clear separation of concerns |
| E2E timing | Hybrid — core flows spec-first, edge cases post-impl | Industry best practice (ATDD), avoids E2E bloat |
| Fix loop | Debugger → fix plan → lead verify → fix → retest | Prevents blind fix degradation |
| Max retries | 3 failures → user escalation | Industry standard (3-5 retries) |
| Framework | qa-tester auto-detection (Playwright/WebdriverIO/etc) | Already implemented, platform-aware |
| Test author | Implementation agent writes tests, separate verification | Simpler team composition |
| Report format | Concise — pass/fail count + failed test name + error | Minimal noise |
| TDD vs E2E-first | E2E tests satisfy the "test-first" requirement; unit tests optional during impl | E2E covers user flows; unit tests complement |

### 2. team-setup — Procedural E2E Pipeline

Replace the `## Workflow Per Task` section in the **team-rules.md template** (generated by Phase 5.1 of team-setup/SKILL.md) with the new 6-phase procedural pipeline. Also update the `## Verification Gate` section in the same template.

**Clarification**: This changes the *template content* that team-setup generates into `.claude/rules/team-rules.md`, NOT the team-setup SKILL.md's own Phase structure (Phases 1-7 of the skill remain unchanged).

**TDD/E2E relationship**: The 6-phase pipeline supersedes the unit-test-first workflow for feature tasks. E2E tests are the primary pre-implementation tests and satisfy the "test-first" requirement from tdd-sdd.md. Unit tests remain optional during implementation (Phase 3). Add a note in the generated team-rules.md: "E2E tests fulfill the TDD test-first mandate. Unit tests are complementary, not required before implementation."

#### Current (Declarative — in team-rules.md template)
```
1. Read spec
2. Write failing test (RED)
3. Implement code (GREEN)
4. Refactor
5. Run full test suite
6. Type check + lint
7. Submit
```

#### New (Procedural E2E Pipeline — replaces above in team-rules.md template)

```
## Task Execution Pipeline (Mandatory)

### Precondition: E2E Infrastructure
- If no E2E framework is detected (no config file, no e2e/ directory):
  Set up E2E infrastructure first (per project-setup E2E scaffolding template).
- If E2E framework exists: proceed to Phase 1.

### Phase 1: Spec Confirmation
- Read spec/PRD. If none exists, create one first.
- Identify core user flows (based on acceptance criteria).

### Phase 2: E2E Test — Core Flows (Pre-Implementation)
- Write E2E tests for identified core user flows.
- Run tests → MUST confirm FAIL (RED).
- Commit E2E test files (creates timestamp evidence for verification gate).
- Report test file paths and failure results to team lead.
  Example: "e2e/flows/login.spec.ts — 3 tests, 3 failed (expected)"

### Phase 3: Implementation (GREEN)
- Implement code to pass the E2E tests.
- Unit tests may be added freely during implementation.

### Phase 4: E2E Verification
- Run FULL E2E test suite.
- Generate concise report:
  ✅ passed: 12  ❌ failed: 1  ⏱ duration: 8.3s
  FAIL: e2e/flows/checkout.spec.ts > "order confirmation after payment"
  Error: Expected status 200, got 500
- ALL PASS → Phase 6
- ANY FAIL → Phase 5

### Phase 5: Fix Loop (max 3 rounds)
Round N:
  1. Debugger: Analyze failure root cause + generate report
  2. Debugger: Verify whether the TEST ITSELF is correct
     - If test is wrong → propose test fix (not code fix)
     - If code is wrong → proceed to step 3
  3. Implementation agent: Write fix plan
  4. Team lead: Review fix plan → approve/reject
     - If rejected: implementation agent revises plan and resubmits
       (max 2 plan revisions per round, does NOT count toward 3-round limit)
     - If approved: proceed to step 5
  5. Implementation agent: Execute approved fix
  6. Re-run E2E suite → PASS → Phase 6, FAIL → Round N+1

  After 3 failures:
  - Generate escalation report with failure history
  - Set task status → BLOCKED
  - Escalate to user

### Phase 6: Edge Cases + Polish
- Add edge case / error scenario E2E tests.
- Refactor code (keep tests green).
- Run full test suite (final).
- Report to team lead with results.

### TDD/E2E Relationship
E2E tests fulfill the TDD test-first mandate. Unit tests are complementary,
not required before implementation. The tdd-sdd.md "write failing test" step
is satisfied by Phase 2 E2E test writing.
```

#### Verification Gate (Updated — replaces existing gate in team-rules.md template)

```
## Verification Gate (MUST Pass Before Acceptance)
1. ✅ Spec exists and is current
2. ✅ Core flow E2E tests committed BEFORE implementation (evidence: E2E test commit precedes impl commit)
3. ✅ Full E2E suite PASS (report attached)
4. ✅ Edge case E2E tests added
5. ✅ Full regression suite PASS
6. ✅ Type check + lint PASS
7. ✅ Code polish complete
```

#### Fix Loop Report Format

```
## Fix Loop Report — Round {N}
- Failed test: {test file}:{test name}
- Error: {error message}
- Root cause: {debugger analysis}
- Test validity: VALID / INVALID / N/A (if invalid: proposed test fix)
- Fix plan: {description}
- Lead approved: yes/no (if no: revision count)
- Result after fix: PASS/FAIL
```

### 3. TaskCompleted Hook — E2E Verification Gate

**Location**: Update the TaskCompleted hook template in `project-setup/setup-templates.md` Section B (settings.json hooks) AND in `team-setup/SKILL.md` Phase 6 hook generation.

#### Current
```json
"TaskCompleted": [{
  "hooks": [{
    "type": "prompt",
    "prompt": "Verify: 1. Were tests written BEFORE implementation?..."
  }]
}]
```

#### New

```json
"TaskCompleted": [{
  "hooks": [{
    "type": "prompt",
    "prompt": "A teammate reported task completion. Perform these checks IN ORDER:

STEP 1: E2E Evidence Check
- Does the completion message include an E2E report?
- Expected format: ✅ passed: N  ❌ failed: N  ⏱ duration: Ns
- If missing → REJECT: 'Attach E2E test execution results'

STEP 2: E2E Result Validation
- If failed > 0 → REJECT: 'E2E test failures exist. Enter Fix Loop'
- If passed == 0 → REJECT: 'No E2E tests found'

STEP 3: Core Flow Coverage
- Do E2E tests cover the spec's acceptance criteria?
- If missing flows → REJECT: 'Core flow {flow_name} E2E missing'

STEP 4: Regression Check
- Full test suite results (existing tests not broken)
- If regression failures → REJECT: 'Regression test failures'

STEP 5: Standard Checks
- Type check PASS
- Lint PASS
- Code polish complete

ALL PASS → ACCEPT the task
ANY REJECT → Send rejection with reason + required action"
  }]
}]
```

#### Fix Loop Auto-Trigger

On REJECT, team lead sends to implementation agent:

```
Fix Loop Round {N}/3 triggered.
Reason: {rejection reason}
Action required:
1. Request failure analysis from debugger agent
2. Debugger verifies: is the TEST correct or is the CODE wrong?
3. Write fix plan based on analysis
4. Report fix plan to team lead (me) for approval
   - If plan is rejected, revise and resubmit (max 2 revisions)
5. After approval, execute fix
6. Re-run full E2E suite and report completion again

Failure count: {N}/3. Escalation to user after 3 failures.
```

#### Escalation Message (after 3 failures)

```
⚠️ E2E Fix Loop Failed 3 Times — User Intervention Required
Task: {task name}
Failed test: {test file}:{test name}
Attempts:
  Round 1: {root cause} → {fix applied} → test validity: {VALID/INVALID/N/A} → FAIL
  Round 2: {root cause} → {fix applied} → test validity: {VALID/INVALID/N/A} → FAIL
  Round 3: {root cause} → {fix applied} → test validity: {VALID/INVALID/N/A} → FAIL
Action needed: User must decide direction
```

## Validation Against Industry Standards

| Aspect | Our Design | Industry Pattern | Status |
|--------|-----------|-----------------|--------|
| Spec-first E2E | Phase 1→2: spec → E2E before code | ATDD / Specification by Example | ✅ Aligned |
| Hybrid coverage | Core flows first, edge cases after | "Prioritize critical workflows" | ✅ Aligned |
| Fix loop structure | Debugger → plan → lead verify → fix | Evaluator Reflect-Refine Loop (AWS) | ✅ Aligned |
| Test validity check | Debugger verifies test correctness | "Don't blindly fix code" caveat | ✅ Mitigated |
| Plan rejection handling | Max 2 revisions per round, separate from failure count | Prevents rejection deadlock | ✅ Addressed |
| Retry limit | 3 attempts + escalation | 3-5 retries industry standard | ✅ Aligned |
| Specialized agents | Debugger analyzes, implementer fixes | Council of Sub Agents pattern | ✅ Aligned |
| E2E infra precondition | Auto-scaffold if missing | Shift-left testing readiness | ✅ Addressed |
