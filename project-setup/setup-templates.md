### Session Protocol Section
```markdown
## Session Protocol

**Start:**
1. Read `docs/progress.txt` — what happened last session
2. Read `docs/features-list.md` — identify next task
3. Read relevant spec (if exists) before coding
4. Plan before code — use Plan Mode (Shift+Tab ×2) for exploration, then switch to execution

**Plan Mode (Shift+Tab ×2):**
- Read-only mode — no code changes allowed, safe for exploration
- Use for: investigating bugs, understanding architecture, planning features
- Confirm plan → Shift+Tab to switch to execution mode
- Complex architecture decisions: add "ultrathink" to prompt for deeper reasoning

**End:**
1. Commit all changes
2. Update `docs/progress.txt`
3. Update `docs/features-list.md` status
4. Update `docs/changelog/CHANGELOG.md`
5. Update specs if code changed behavior

**Stuck (2+ failed fix attempts):**
1. STOP — do not attempt a third time
2. Commit current state
3. Record in progress.txt: what tried, why failed, which files
4. /clear → new session → different approach

**Periodic Review:**
- Run `/insights` monthly to analyze usage patterns and identify workflow improvements
```

### Advanced Hook Patterns

#### Async Hook (background execution)
Use `async: true` for heavy analysis that shouldn't block Claude's response:
```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "node .claude/scripts/quality-report.js",
            "async": true,
            "timeout": 60
          }
        ]
      }
    ]
  }
}
```
Results are written to `.claude/reports/` — check on next session start.

#### HTTP Hook (remote policy validation)
Send tool call data to a remote server for centralized team policy enforcement:
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash|Write",
        "hooks": [
          {
            "type": "http",
            "url": "https://policy.internal/validate",
            "method": "POST",
            "timeout": 5
          }
        ]
      }
    ]
  }
}
```
Server returns 200 = allow, non-200 = block. Use for team-wide coding standards enforcement.

#### MCP Tool Matching Hook (audit MCP operations)
Hook into MCP tool calls using `mcp__<server>__<tool>` pattern matching:
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "mcp__plugin_serena_serena__replace_symbol_body|mcp__plugin_serena_serena__replace_content",
        "hooks": [
          {
            "type": "command",
            "command": "echo \"[MCP Edit] $(date +%H:%M:%S)\" >> .claude/logs/mcp-edits.log"
          }
        ]
      }
    ]
  }
}
```
Creates audit trail for MCP-driven code modifications.

#### PreToolUse Security Gate (block dangerous commands)
PreToolUse is the only blocking hook — exit code 2 blocks the tool call:
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo $CLAUDE_TOOL_INPUT | node -e \"const s=require('fs').readFileSync('/dev/stdin','utf8'); const blocked=['rm -rf /','DROP TABLE','--force','--no-verify','chmod 777']; const cmd=JSON.parse(s).command||''; if(blocked.some(b=>cmd.includes(b))){console.error('BLOCKED: dangerous command'); process.exit(2)}\""
          }
        ]
      }
    ]
  }
}
```
Customize the `blocked` array for project-specific dangerous patterns.

### How the Auto-Handoff System Works

```
[Coding...]
  ↓ Context reaches compaction threshold
  ↓ PreCompact hook fires → prompt: "Write .claude/handoff.md"
  ↓ Claude writes handoff with: completed, pending, decisions, lore, next steps
  ↓ Compaction or /clear happens
  ↓ SessionStart hook fires (matcher: "compact" or "clear")
  ↓ Hook outputs handoff.md content → injected into new session context
  ↓ Seamless continuation — no manual intervention needed
```

User only needs to type `/clear` — everything else is automatic.

### Alternative: Shell-script based hooks (for complex logic)

session-start.sh:
```bash
#!/bin/bash
# Auto-load handoff if exists (after /clear or compaction)
HANDOFF=".claude/handoff.md"
if [ -f "$HANDOFF" ]; then
  echo "=== Session Handoff Found ==="
  cat "$HANDOFF"
  echo ""
fi

PROGRESS_FILE="docs/progress.txt"
if [ -f "$PROGRESS_FILE" ]; then
  echo "=== Progress ==="
  head -25 "$PROGRESS_FILE"
  echo ""
fi

echo "=== Git ==="
git branch --show-current 2>/dev/null
git status --short 2>/dev/null
echo ""
echo "=== Recent Commits ==="
git log --oneline -5 2>/dev/null
echo ""
echo "=== Pending ==="
grep -c 'FAILING\|❌' docs/features-list.md 2>/dev/null && echo " items remaining" || echo "no features-list.md"
```

session-stop.sh:
```bash
#!/bin/bash
echo "=== Session End Check ==="

# Check handoff file
HANDOFF=".claude/handoff.md"
if [ -f "$HANDOFF" ]; then
  FILE_TIME=$(stat -c %Y "$HANDOFF" 2>/dev/null || stat -f %m "$HANDOFF" 2>/dev/null)
  NOW=$(date +%s)
  DIFF=$((NOW - FILE_TIME))
  if [ "$DIFF" -gt 1800 ]; then
    echo "[!] handoff.md not updated in 30+ minutes."
  else
    echo "[OK] handoff.md recently updated."
  fi
else
  echo "[!] No handoff.md — session state may be lost on /clear."
fi

# Check uncommitted changes
CHANGES=$(git status --porcelain 2>/dev/null | wc -l)
if [ "$CHANGES" -gt 0 ]; then
  echo "[!] Uncommitted changes: ${CHANGES} files"
  git status --short 2>/dev/null
else
  echo "[OK] All changes committed."
fi
```

### tdd-sdd.md
```markdown
# Development Methodology: SDD + TDD

This project follows Spec-Driven Development (SDD) + Test-Driven Development (TDD).
All contributors (including Agent Team teammates) MUST follow these rules.

## SDD: Spec-Driven Development

### Mandatory Workflow
1. **Spec first**: Read or create feature spec before ANY code changes
2. **No code without spec**: If no spec exists → create one → get approval → then code
3. **Spec-code sync**: After modifying code, update matching specs
4. **Acceptance criteria**: Every spec must have explicit, testable acceptance criteria

### Spec Location
- Feature specs: `docs/specs/features/`
- API specs: `docs/specs/api/`
- DB specs: `docs/specs/db/`

## TDD: Test-Driven Development

### Red-Green-Refactor Cycle (Mandatory)
1. **RED**: Write failing test that describes expected behavior
2. **GREEN**: Write minimal code to pass the test
3. **REFACTOR**: Clean up code while keeping tests green

### Testing Requirements
1. **Test before implementation**: Write tests BEFORE writing feature code
2. **Regression tests**: Run ALL existing tests before marking work complete
3. **Edge cases**: Test error scenarios, boundary conditions, null/empty inputs
4. **Test quality**: Tests must verify behavior, not just cover lines
5. **Full suite pass**: `{{TEST_COMMAND}}` must pass with 0 failures before completion

### Bug Fix Testing (Critical)
When fixing bugs:
1. Write a test that reproduces the bug FIRST
2. Verify the test FAILS (proving the bug exists)
3. Implement the fix
4. Verify the reproduction test PASSES
5. Run FULL test suite — no regressions allowed
6. Verify related functionality still works

### Forbidden Patterns
- ❌ Writing code before tests
- ❌ Marking task complete with failing tests
- ❌ Skipping regression test suite (running only new tests)
- ❌ Writing tests that always pass (tautological tests)
- ❌ Deleting failing tests instead of fixing code
- ❌ Mocking everything — test real behavior where possible

## Verification Gate (Mandatory Before Completion)

Every task MUST pass ALL checks before being marked complete:
1. ✅ Spec exists and is up-to-date
2. ✅ Tests written BEFORE code (TDD)
3. ✅ All new tests pass
4. ✅ Full test suite passes (regression): `{{TEST_COMMAND}}`
5. ✅ Type check passes: `{{TYPECHECK_COMMAND}}`
6. ✅ Lint passes: `{{LINT_COMMAND}}`
7. ✅ Acceptance criteria from spec verified

If ANY check fails → fix before marking complete. No exceptions.
```

- Test teammate (Sonnet): tests/ only

**Multi-Perspective Review** (code review, security audit):
- Lead (Opus): synthesize findings
- Security reviewer (Sonnet): auth, injection, data validation
- Performance reviewer (Sonnet): queries, memory, latency
- Test coverage reviewer (Sonnet): edge cases, missing tests

**Adversarial Debug** (complex bugs, investigations):
- Lead (Opus): moderate debate, synthesize consensus
- N hypothesis investigators (Sonnet): each argues different root cause
- Instruction: "Challenge each other's theories like a scientific debate"

**Architecture Decision** (tech choices, design):
- Lead (Opus): produce final ADR document
- N advocates (Sonnet): each argues for different approach
- Instruction: "Challenge each other on performance, cost, complexity, migration effort"

### Team Spawn Best Practices
Always include in spawn prompts:
1. `"Use delegate mode — only coordinate, don't implement yourself"`
2. `"ONLY touch files in [assigned directories]"` — per teammate
3. `"Require plan approval before making any changes"`
4. `"Use Sonnet for each teammate"`
5. Do NOT duplicate CLAUDE.md content in spawn prompts (auto-loaded)

### Team Session Protocol
- `/compact` at 70-80% context (before 95% auto-compression)
- Shut down idle teammates immediately (they consume tokens)
- Always: shutdown all teammates → then clean up team (never reverse)
- In-process teammates cannot be resumed after `/resume` — re-spawn needed

### Team Cost Awareness
- 3 teammates ≈ 3-4x token cost vs single session
- 5 teammates ≈ 7x token cost
- Optimize: Sonnet teammates, concise spawn prompts, prompt cleanup

## I. CI/CD Headless Mode (Optional)

Automate code review and quality checks in CI pipelines using Claude Code headless mode.

### GitHub Actions — PR Auto-Review
```yaml
# .github/workflows/claude-review.yml
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code
      - name: Run Code Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude -p "Review this PR's changes. Check for: 1) Logic errors 2) Security issues 3) Test coverage gaps 4) Convention violations. Output as markdown." \
            --format text \
            --allowedTools Read,Grep,Glob \
            > review.md
      - name: Post Review Comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const review = fs.readFileSync('review.md', 'utf8');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: review
            });
```

### Headless Mode Best Practices
- **Always restrict tools**: `--allowedTools Read,Grep,Glob` (never allow `Bash` in CI)
- **Output formats**: `text` (human), `json` (machine), `stream-json` (real-time)
- **Cost control**: Use with Haiku for simple checks, Sonnet for reviews
- **Use cases**: PR review, changelog generation, accessibility audits, tech debt scoring

## J. PostToolUse Format Commands by Tech Stack

Replace `{{FORMAT_COMMAND}}` in the PostToolUse hook based on detected tech stack:

| Tech Stack | Format Command |
|-----------|----------------|
| Node.js / TypeScript | `npx prettier --write "$CLAUDE_FILE_PATH" 2>/dev/null \|\| true` |
| Python | `ruff format "$CLAUDE_FILE_PATH" 2>/dev/null \|\| black "$CLAUDE_FILE_PATH" 2>/dev/null \|\| true` |
| Go | `gofmt -w "$CLAUDE_FILE_PATH" 2>/dev/null \|\| true` |
| Rust | `rustfmt "$CLAUDE_FILE_PATH" 2>/dev/null \|\| true` |
| Java | `google-java-format -i "$CLAUDE_FILE_PATH" 2>/dev/null \|\| true` |
| None detected | `echo "No formatter configured"` |

**Note:** If project uses ESLint with `--fix`, consider: `npx eslint --fix "$CLAUDE_FILE_PATH" 2>/dev/null; npx prettier --write "$CLAUDE_FILE_PATH" 2>/dev/null || true`
